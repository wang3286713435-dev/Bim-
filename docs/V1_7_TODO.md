## 当前结论

`v1.6` 已经形成一套可用、可维护、可继续迭代的 BIM 招采监控体系。  
`v1.7` 暂时不追求“大功能新增”，而是进入：

1. 稳定性收口
2. 问题归档
3. 观察期复盘
4. 有价值需求的精选进入

当前生产基线：

- 生产版本：`1.6.13`
- AI 主链路：`OpenClaw(main + --local)`
- AI 备用链路：`OpenRouter` 已接通
- 旧版 `legacy` 样本：`0`
- 当前 `fallbackRows`：`0`
- 主库仅保留当前招采监控来源：`95` 条

一句话判断：

> `v1.6` 现在最需要的不是继续加能力，而是观察真实使用、收集真实问题，然后把 `v1.7` 做成一个更稳、更清晰、更好维护的版本。

## 观察期原则

观察期内默认遵循：

1. 不急着加新数据源
2. 不急着攻 `CEB` 深详情
3. 不急着做大前端改版
4. 不急着切 MySQL
5. 新想法先记录，先不立即开发
6. 新 bug 先归档到本文件，再决定是否进入 `P0 / P1`

## 观察期重点看什么

### 1. AI 稳定性

- 是否再次出现大面积 `规则回退`
- `OpenClaw` 是否有明显波动
- `OpenRouter fallback` 是否真的在异常时接管
- AI 耗时是否在高峰期明显变慢

### 2. 来源健康

- 哪些来源最容易失败
- 失败是列表、详情还是 AI 分析阶段
- 代理池降级是否符合预期
- 是否还会出现“看起来健康，实际被目标站封禁”的情况

### 3. 字段质量

- 单位、预算、联系人、电话、截止时间是否仍有明显缺口
- 某些来源是否反复出现同类脏字段
- 详情补强是否还有“看着成功，实际字段没价值”的情况

### 4. 前端体验

- 哪些信息密度仍然偏高
- 哪些模块用户最常看
- 哪些提示语仍容易引起误解
- 哪些地方属于“不是坏，但还不够顺手”

## 我对 1.7 的建议

`v1.7` 最好做成一个“小而硬”的版本，而不是“功能更多”的版本。

更具体地说：

1. 让我们知道系统什么时候坏、为什么坏、坏到什么程度
2. 让真实招采源和历史噪声永远分开
3. 让 AI、代理池、详情补强都有更清楚的健康口径
4. 让后续新功能建立在稳态之上，而不是建立在“凑合能跑”之上

## 进入 v1.7 的条件

建议满足以下条件再正式开工：

1. `v1.6` 观察期内未出现新的主链路级故障
2. 主力来源健康状态基本稳定
3. 观察期内的 bug / 想法已经收集到一定数量
4. 我们能明确分出：
   - 必修项
   - 可选项
   - 延后项

到那时再正式进入 `v1.7`，节奏会更稳。
